# v0.3.1 — Stornorechnung (Credit-Note) ZUGFeRD Audit

Audit of the credit-note (Stornorechnung) ZUGFeRD-XML output shipped in v0.3
against the EN-16931 CII binding (UN/CEFACT XSD). Scope: structural compliance
of the rendered XML; sign handling for monetary totals; preceding-invoice
reference (BG-3 / BT-25 / BT-26).

Reference files inspected:
- `sidecar/templates/zugferd-en16931.xml.j2`
- `sidecar/invoice/zugferd.py`
- `src/lib/db/invoices.ts`
- `src/lib/sidecar/invoice.ts`

## Findings

### 🔴 InvoiceReferencedDocument was misplaced inside `ApplicableHeaderTradeSettlement`

The UN/CEFACT CII schema (`HeaderTradeSettlementType`) defines an explicit
child sequence. `InvoiceReferencedDocument` (BG-3, holding BT-25/BT-26) must
appear **after** `SpecifiedTradeSettlementHeaderMonetarySummation`, near the
end of the settlement block. v0.3 placed it directly after
`InvoiceCurrencyCode`, before `SpecifiedTradeSettlementPaymentMeans` /
`ApplicableTradeTax` / `SpecifiedTradeSettlementHeaderMonetarySummation`,
which is schema-invalid ordering and would be rejected by a strict XSD
validator (and by erechnungs-validator.de's KoSIT validator chain).

The element placement *inside* `ApplicableHeaderTradeSettlement` is correct
(it is **not** in `ApplicableHeaderTradeAgreement` — that is OASIS UBL
territory; CII puts it in Settlement). Only the position within Settlement
was wrong.

- Was: `sidecar/templates/zugferd-en16931.xml.j2:136-143` (between
  `InvoiceCurrencyCode` and `SpecifiedTradeSettlementPaymentMeans`).
- Now: emitted after `SpecifiedTradeSettlementHeaderMonetarySummation`,
  immediately before the closing `</ram:ApplicableHeaderTradeSettlement>`.

### ✓ `qdt:` namespace declared at root

`xmlns:qdt="urn:un:unece:uncefact:data:standard:QualifiedDataType:100"` is
declared on `<rsm:CrossIndustryInvoice>` (line 8). Required because
`FormattedIssueDateTime` uses `qdt:DateTimeString` (not `udt:`).

### ✓ `BT-3` `TypeCode` = `381` for credit notes

`zugferd-en16931.xml.j2:21` — `{% if invoice.isCreditNote %}381{% else %}380{% endif %}`.

### ✓ Element shape of `InvoiceReferencedDocument`

```xml
<ram:InvoiceReferencedDocument>
  <ram:IssuerAssignedID>{{ corrects.number }}</ram:IssuerAssignedID>
  <ram:FormattedIssueDateTime>
    <qdt:DateTimeString format="102">YYYYMMDD</qdt:DateTimeString>
  </ram:FormattedIssueDateTime>
</ram:InvoiceReferencedDocument>
```

Matches the EN-16931 CII binding for BT-25 + BT-26 exactly. `format="102"`
is CCYYMMDD per UN/EDIFACT 2379 — correct.

### ✓ Monetary totals are negative end-to-end for credit notes

Sign propagation verified:

- Per-line `LineTotalAmount`: template multiplies by `-1` when
  `invoice.isCreditNote` (`zugferd-en16931.xml.j2:76`).
- `vat_groups` in `sidecar/invoice/zugferd.py:54-67` receives a `sign`
  argument (`-1` for credit notes, line 99) which negates each line before
  bucketing, so `BasisAmount` and `CalculatedAmount` per
  document-level `ApplicableTradeTax` are negative.
- Header `SpecifiedTradeSettlementHeaderMonetarySummation` consumes
  `invoice.subtotal`, `invoice.vatAmount`, `invoice.total` straight from
  the payload. These are stored negated in the DB:
  `src/lib/db/invoices.ts:281,295-296,324,346-347` apply
  `sign = isCreditNote ? -1 : 1` on both insert and update, so the values
  arriving at the template are already negative.
- Result: `LineTotalAmount`, `TaxBasisTotalAmount`, `TaxTotalAmount`,
  `GrandTotalAmount`, `DuePayableAmount` are all rendered negative, which
  is what EN-16931 BR-CO-10..BR-CO-16 expect for a `TypeCode 381` document.

### ✓ `SpecifiedTradePaymentTerms` absent for credit notes

`zugferd-en16931.xml.j2:182` — guarded by `{% if not invoice.isCreditNote and (invoice.paymentTerms or invoice.dueDate) %}`.
Credit notes have no due date by definition; suppressing the block is correct.
(BR-CO-25 only requires `DueDateDateTime` *or* `Description` if the block
exists; omitting the block entirely is permitted.)

### ✓ `factur-x.generate_from_binary` level

`sidecar/invoice/pdf.py` passes `payload.profile or "en16931"` unchanged.
The factur-x library treats the level as the embedded-XML filename
selector + metadata, not as a content validator, so credit notes work
under any level. No change needed.

### ✓ Per-line `ApplicableTradeTax` (`CategoryCode` / `RateApplicablePercent`)

Uses the same Reverse-Charge / Kleinunternehmer / Standard branching as
regular invoices. Credit notes inherit the original invoice's tax regime
via the snapshot on creation (`src/lib/db/invoices.ts:386-406`). Correct —
a credit note for a 19% invoice must use 19% on every line so the
negative tax basis offsets the original positive one in the buyer's
books.

### ⚠️ `BT-26` issue date precision

`invoice.correctsInvoice.issueDate` is a unix-seconds timestamp formatted
as `YYYYMMDD` via `_date_xml` using `datetime.fromtimestamp(..., tz=UTC)`.
For an invoice issued near midnight in CET/CEST this can shift the date
one day earlier when rendered in UTC. Same caveat exists for regular
invoices (`BT-2 IssueDateTime`), so this is not credit-note-specific and
is out of scope for v0.3.1. Tracking for a future fix would be welcome
(use the user's local TZ or a date-only column).

### ⚠️ Credit-note PDF filename prefix

Not part of the XML audit, but worth noting: the output path generator
`buildOutputPath` (`src/lib/sidecar/invoice.ts`) uses the invoice's
`number` field, which already carries the credit-note prefix (e.g.
`RE-2026-0007-STORNO` or whatever the configured format is). No XML
impact; visual only.

## Fixes applied

```diff
--- a/sidecar/templates/zugferd-en16931.xml.j2
+++ b/sidecar/templates/zugferd-en16931.xml.j2
     <ram:ApplicableHeaderTradeSettlement>
       <ram:InvoiceCurrencyCode>EUR</ram:InvoiceCurrencyCode>
-      {% if invoice.isCreditNote and invoice.correctsInvoice %}
-      <ram:InvoiceReferencedDocument>
-        <ram:IssuerAssignedID>{{ invoice.correctsInvoice.number }}</ram:IssuerAssignedID>
-        <ram:FormattedIssueDateTime>
-          <qdt:DateTimeString format="102">{{ invoice.correctsInvoice.issueDate | date_xml }}</qdt:DateTimeString>
-        </ram:FormattedIssueDateTime>
-      </ram:InvoiceReferencedDocument>
-      {% endif %}
       {% if company.bankIban %}
       ...
       </ram:SpecifiedTradeSettlementHeaderMonetarySummation>
+      {% if invoice.isCreditNote and invoice.correctsInvoice %}
+      <ram:InvoiceReferencedDocument>
+        <ram:IssuerAssignedID>{{ invoice.correctsInvoice.number }}</ram:IssuerAssignedID>
+        <ram:FormattedIssueDateTime>
+          <qdt:DateTimeString format="102">{{ invoice.correctsInvoice.issueDate | date_xml }}</qdt:DateTimeString>
+        </ram:FormattedIssueDateTime>
+      </ram:InvoiceReferencedDocument>
+      {% endif %}
     </ram:ApplicableHeaderTradeSettlement>
```

Single move; no semantic change beyond child-element ordering. All other
template branches, sign math, and namespace declarations were already
correct.

## Outstanding (needs real PDF round-trip)

These cannot be confirmed by code review alone — they require generating
a real Stornorechnung PDF and uploading it to
https://erechnungs-validator.de (KoSIT validator + Mustang test suite):

1. **Schema validation** of the relocated `InvoiceReferencedDocument` —
   confirm the CII XSD now accepts the document. Expected: green.
2. **Schematron rules**:
   - `BR-CO-3`: Value-added-tax-point-date and date-of-issue must follow
     ISO 8601 — covered, but verify on a generated file.
   - `BR-CO-10` / `BR-CO-13` / `BR-CO-15` / `BR-CO-16`: sum-of-line ==
     `TaxBasisTotalAmount`, etc., with negative values. Spot-check on a
     two-line credit note that the math still validates with negatives.
   - `BR-55` (Preceding Invoice reference must have an Issuer Assigned
     ID) — covered.
3. **PDF/A-3 conformance** of the credit note (factur-x library does
   this; verify no regression vs. regular invoices).
4. **Buyer-side ingest**: Try importing the generated XML into a known
   consumer (e.g. DATEV Belegtransfer test, or the Mustang viewer) and
   confirm the credit note is recognized as such (TypeCode 381 → cancels
   the referenced invoice).
