# v0.4 Release Audit

Audit performed on `release/v0.4` against merge base `f1bbce9` (last `main`),
covering six features merged in parallel: PR #23 (Vorlage-aus-Rechnung),
PR #24 (Storno-XML fix), PR #25 (PDF-Themes), PR #26 (granular restore),
PR #27 (Reverse-Charge Drittland), PR #28 (AES-256-GCM backup encryption).

## Summary

- **9 ✓ green**, **1 ⚠️ minor concern**, **2 🔴 broken** — both fixed inline.
- Go/No-Go: **Go** after the two fixes below. No remaining release blockers.

## Findings

### 🔴 1. Stale `CURRENT_SCHEMA = 4` in `backup.rs` — rejects every modern backup

`src-tauri/src/backup.rs:236` (before fix) hard-coded `const CURRENT_SCHEMA: u32 = 4;`
in `stage_restore`. Every backup created by this build writes
`db_schema_version = 9` into the manifest (via `CURRENT_DB_SCHEMA_VERSION` from
`Settings.svelte:24`). The check
`if parsed.db_schema_version > CURRENT_SCHEMA` would fail with
"Backup-Schema 9 ist neuer als die App-Version (Schema 4). Bitte App aktualisieren."
on the very first restore attempt — i.e. backup/restore is *completely* broken
in v0.4 as-shipped without this fix.

**Fixed:** bumped to `9` and added a comment pointing at the two sibling
constants that must stay in sync.

### 🔴 2. Stale `CURRENT_DB_SCHEMA_VERSION = 8` in `Settings.svelte`

`src/routes/Settings.svelte:24` declared `CURRENT_DB_SCHEMA_VERSION = 8`. This
constant is written into the backup manifest by `bundle_backup`. PR #27 added
migration 0008 (`PRAGMA user_version = 9`, registered as `version: 9` in
`lib.rs`) but did not bump the frontend mirror. Result: backups created on v0.4
would have stamped `db_schema_version = 8` despite running schema 9 — minor on
its own, but combined with finding 1 it masks the bug.

**Fixed:** bumped to `9`.

### ⚠️ 3. Doc/code mismatch in `crypto.rs` magic-byte comment

`src-tauri/src/crypto.rs:3` describes the magic prefix as
`"ZETTEL-ENC-1\n" (12 bytes — magic + version + newline)`, but the actual
constant is `b"ZETTEL-ENC-1"` (12 bytes, **no** newline — the literal is 12
ASCII chars). Behaviour is consistent (encrypt and `is_encrypted` use the same
constant), so this is purely a comment bug. Not fixed — not worth touching
crypto code for a comment.

### ✓ 13. Offers untouched by theme/RC

- `offer.html.j2` does not reference `settings`; no risk of undefined access.
- `sidecar/invoice/pdf.py:_render_offer_html_pdf` does not pass `settings` —

- `convertToInvoice` (`offers.ts:425`) maps offer's legacy boolean

  third-country exports from offers are not modelled. Worth a follow-up issue

### ✓ 14. `bundle_backup` / `stage_restore` signature changes propagated

Both Rust commands now take `password: Option<String>`. Frontend callers in
`Settings.svelte:129–135`, `:172–180`, `:206–210` all pass the new arg
(`backupPassword` or `null`). No stale invocation sites elsewhere
(`grep -rn` confirmed only these three).

### ✓ 15. Granular restore — table dependency order

`backup.rs:418–446` deletes children-before-parents (`invoice_items` before
`invoices`, `offer_items` before `offers`, `recurring_invoice_items` before
`recurring_invoices`) inside a `PRAGMA foreign_keys=OFF` block, then inserts
parents-before-children via `tables.iter().rev()`. Logically sound; FK
constraints are disabled inside the transaction so even an out-of-order step
would not block, but the careful ordering keeps the DB consistent if the
transaction is inspected mid-flight.

1. `src-tauri/src/backup.rs:236` — `CURRENT_SCHEMA 4 → 9` + sync comment.
2. `src/routes/Settings.svelte:24` — `CURRENT_DB_SCHEMA_VERSION 8 → 9`.

## Go/No-Go

**Go.** v0.4 ships six independent features with clean cross-PR seams in the
hot zones — schema, sidecar payload, ZUGFeRD template, invoice HTML, Tauri
command registration. The only broken edges were two stale schema-version
constants (one Rust, one TypeScript) that together would have rendered the
backup/restore feature inoperative on v0.4; both are now corrected. The
crypto comment mismatch is cosmetic. Recommend merging `release/v0.4 → main`
after a smoke test on Windows that exercises:
(a) create + restore an encrypted backup,
(b) restore a v0.3 backup (`db_schema_version <= 7`) for migration coverage,
(c) generate one invoice per `reverseChargeType` and validate against
    erechnungs-validator.de,
(d) re-render each invoice in all three themes.
